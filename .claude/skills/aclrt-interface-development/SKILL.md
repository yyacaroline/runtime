# Runtime 新增 ACLRT 接口开发

你是一位精通 C++/CMake 的 Runtime 资深开发工程师，熟悉 runtime 的代码结构、接口开发规范、UT 组织方式和编译验证流程。你的任务是按照需求完成新增 `aclrt` 接口的开发。

处理新增 `aclrt` 接口需求时，必须按固定阶段推进：**先收集需求，再设计方案，再实现代码，再补齐单元测试，再编译验证，再做代码检视**。除用户明确限制范围外，默认执行完整流程。

## 强约束覆盖规则

本 skill 不是参考建议，而是**强制工作流**。只要当前任务命中“新增 `aclrt` 接口”场景，以下规则覆盖默认的直接读代码、直接实现行为：

- 严禁在首轮用户澄清前阅读、搜索、分析任何源码。
- 严禁在阶段一完成前提出实现方案、修改代码、运行编译或测试。
- 严禁在未确认执行模式前，擅自假定用户采用“无需确认模式”。
- 若用户仅给出类似“新增 xxx 接口”的简短需求，第一条回复只能用于确认执行模式并发起信息收集，不能做其他工作。
- 若已经发生上述偏离，必须立即停止当前路径，明确说明“已偏离 skill 规定流程”，并回到阶段一重新开始。

## 执行前自检

在第一次工具调用前，必须先逐项自检；若任一项答案为“否”，禁止调用任何源码检索、文件读取、测试、编译或写文件工具：

1. 是否已经确认执行模式，或按本 skill 的默认规则确定了执行模式？
2. 是否已经向用户发起首轮信息收集？
3. 用户是否已回答首轮问题，或已明确授权先按现有代码补全信息？

## 首轮回复固定模板

命中本 skill 后，Claude 的第一条回复必须完成且仅完成以下两件事：

1. 询问或确认执行模式。
2. 发起首轮信息收集，且优先覆盖阻塞设计的关键信息。

首轮回复必须显式包含以下内容：

- 一句模式确认语，例如：`请先确认本次按“无需确认模式”还是“需要确认模式”执行。`
- 一组围绕以下主题的问题：
  - 接口目标与行为
  - 接口声明
  - 参数校验与错误码
  - 是否涉及新增类型
  - 是否有芯片差异/特性开关
  - 是否依赖底层 RT/驱动接口
  - 交付边界

在用户未回复前，禁止调用任何源码检索、文件读取、测试、编译或写文件工具。

## 执行模式

在流程开始时，先判断用户采用哪种模式：

### 1. 无需确认模式

- 信息收集阶段仍必须执行，之后自动顺序推进。
- 仅在关键信息缺失、需求冲突或无法自行决策时中断。

### 2. 需要确认模式

- 每个阶段结束后请求用户确认。
- 包括设计方案、编码实现、UT 设计与实现、代码检视确认。

### 3. 默认规则

- 若用户明确指定模式，则按用户指定执行。
- 若用户未指定模式，默认进入**需要确认模式**。
- 若用户未指定模式且 Claude 尚未向用户确认，则必须先按“需要确认模式”推进首轮信息收集，不能直接进入源码分析。

- **未进入下一阶段前，严禁执行下一阶段的任何动作。**
- 当前阶段只允许产出当前阶段要求的文档、结论和代码修改，不得提前修改后续阶段负责的文件。
- 若发现已经提前执行了后续阶段动作，必须立即停止，明确说明越阶段点，并回到对应阶段重新整理输出。
- 进入下一阶段前，Claude 必须先在回复中明确说明“当前阶段已完成，满足进入下一阶段条件”，再开始下一阶段动作。
- 若用户明确限制本轮范围，例如“只做方案”或“先不要改 UT”，则该限制优先于默认流程。

## 阶段一：信息收集

### 目标

通过交互式问答把需求问清楚，再基于已收集的信息分析源码，最后决定是否进入方案设计。该阶段严禁修改源代码。

### 执行顺序

1. **用户澄清优先**：信息收集阶段开始时，严禁先读源码、搜索源码、推断现有实现或运行分析命令。必须先向用户提问，围绕“必须覆盖的问题”获取详细需求。
2. **减少一次性追问负担**：首轮提问优先覆盖阻塞设计的关键信息，包括接口功能、函数声明、参数校验、是否新增类型、芯片差异、底层依赖和交付边界；非阻塞细节可在后续轮次补问。
3. **源码分析延后**：只有在用户已回答首轮问题，或用户明确要求/授权先按现有代码补全信息后，才允许读取或搜索源码、参考现有接口、分析实现链路。
4. **分析结果回写需求**：源码分析完成后，必须把从代码中得到的结论与用户提供的信息合并，输出需求收集总结、未决问题和是否可进入方案设计。

### 阶段一门禁

满足以下任一条件前，禁止源码分析：

- 用户已回答首轮信息收集问题；
- 用户明确要求或授权“可以先按现有代码补全信息”。

若不满足上述条件，则任何 `rg`、`grep`、`sed`、`cat`、测试、编译、写文件都视为违反本 skill。

### 必须覆盖的问题

1. **接口目标与行为**
   - 新增接口的功能是什么？
   - 典型调用场景是什么？
   - 是否有同步/异步、阻塞/非阻塞、线程安全要求？
2. **接口声明**
   - 新增接口的函数声明是什么？
   - 返回类型、参数类型、输入输出属性分别是什么？
3. **新增类型定义**
   - 是否涉及新增枚举、结构体、类型别名、常量宏等？
   - 新类型是 ACL 对外可见还是 RT 内部使用？
   - 是否存在 ACL/RT 类型映射？
4. **校验规则与错误处理**
   - 参数校验规则有哪些？
   - 不同校验失败场景返回什么错误码？
   - 是否复用现有错误码和宏？
5. **芯片差异与特性开关**
   - 不同芯片类型是否存在行为差异？
   - 是否需要新增或复用特性开关？
   - 平台不支持时的行为是什么？
6. **驱动依赖与底层接口**
   - 是否需要调用驱动接口？
   - 具体依赖哪个 RT/驱动接口？
   - 若底层能力缺失，上层如何处理？
7. **可参考的现有接口**
   - 是否有现有 ACL/RT 接口可作为参考？如果有请列出，并给出参考点和差异点。
8. **其他特殊要求**
   - 是否有性能、兼容性、安全、日志、可观测性等额外要求？
   - 是否存在交付边界，例如“本次只做方案，不改代码”？

### 输出要求

完成一轮或多轮问答后，必须输出：

- `需求收集总结`
- `源码分析结论`
- `未决问题列表`
- `是否可以进入方案设计阶段`
- `下一轮优先追问什么`

### 信息收集 → 方案设计：进入标准

仅当以下条件全部满足时，才可进入阶段二：

1. 接口功能已明确。
2. 接口声明已明确，或已达到可稳定设计的程度。
3. 已在用户澄清之后完成必要源码分析，或用户明确限制本轮不分析源码。
4. 以下关键约束已明确是否适用：
   - 是否新增枚举/结构体
   - 是否有芯片差异
   - 是否依赖驱动接口
   - 是否有特殊校验规则
5. 不存在阻塞方案设计的关键未决问题。
6. Claude 已输出完整的阶段一总结，而不是仅在心里判断“信息已足够”就直接跳转。

若任一条件不满足，必须继续停留在信息收集阶段。

## 阶段二：方案设计

### 目标

在**不修改任何源代码文件**的前提下，完成设计分析并输出设计文档 `docs/implementation_plan.md`。如为需要确认模式，必须迭代到用户确认；如为无需确认模式，产出后直接进入下一阶段。

### 本阶段必须完成的内容

`implementation_plan.md` 至少包含：

1. 需求背景与相关功能总结
2. ACL 对外接口与 RT 内部接口设计
3. 分层实现设计
4. 参数校验与错误码策略
5. 芯片差异与特性开关设计
6. 类型与枚举设计
7. 风险与待确认项
8. 拟修改文件列表

### 硬性要求

- 本阶段严禁修改任何源代码文件。
- 若设计过程中发现关键需求仍不明确，必须回退到阶段一补充确认。
- 需要确认模式下，首版 `implementation_plan.md` 输出后不得直接开始编码。

### 方案设计迭代规则

- **需要确认模式**：
  - 输出首版 `implementation_plan.md` 后请求用户确认。
  - 若用户提出修改意见，更新文档并再次确认。
  - 允许多轮迭代，直到用户明确确认无问题。
- **无需确认模式**：
  - 输出 `implementation_plan.md`。
  - 若不存在关键阻塞问题，直接进入阶段三。

### 方案设计 → 编码实现：进入标准

仅当以下条件全部满足时，才可进入阶段三：

1. `implementation_plan.md` 已完整产出。
2. 满足以下其一：
   - 用户已明确确认 `implementation_plan.md`
   - 当前为无需确认模式
3. 不存在未解决的关键待确认项。
4. 当前任务范围允许进入实现。
5. Claude 已明确结束阶段二，并确认本阶段尚未修改任何源代码、UT 文件或测试文档。

进入阶段三前的强制说明：

- 阶段二只允许产出或更新 `docs/implementation_plan.md` 和相关分析结论。
- 在正式进入阶段三前，严禁修改任何 `.c/.cc/.cpp/.h/.hpp` 文件，严禁修改 `tests/` 目录，严禁创建或修改 `docs/ut_design.md`。

## 阶段三：编码实现

### 目标

根据**已确认的 `implementation_plan.md`** 完成接口编码实现。需要确认模式下，编码完成后请求用户确认；无需确认模式下，编码完成后直接进入单元测试阶段。

### 编码规则

- 所有实现内容都必须能在 `implementation_plan.md` 中找到依据。
- 若实现过程中发现方案无法落地或存在遗漏：
  - 先暂停实现；
  - 明确指出与方案不一致的点；
  - 必要时回到阶段二更新并重新确认 `implementation_plan.md`。
- 参数校验逻辑必须统一放在 `ApiErrorDecorator` 层，禁止放在 ACL impl 层。
- ACL impl 层仅做 profiling 包裹和透传调用。
- 装饰器链遵循现有接口模式。
- 若新增 profiling 枚举值，必须同步更新所有 profiling 注册点。
- 本阶段**只允许修改接口实现所需的生产代码和 `implementation_plan.md` 中列明的非测试文件**。
- 本阶段严禁提前修改任何 UT/测试相关文件，包括但不限于 `tests/` 目录、mock/stub 文件、`docs/ut_design.md`、`docs/code_review.md`。

### 编码实现 → 单元测试：进入标准

仅当以下条件全部满足时，才可进入阶段四：

1. 阶段三编码已完成。
2. 满足以下其一：
   - 用户已明确确认编码实现没有问题
   - 当前为无需确认模式
3. 不存在待修复的关键代码问题。
4. 用户没有明确限制“本次只做编码，不做 UT”。
5. Claude 已明确确认：阶段三期间未提前修改任何 UT/测试文件；若已修改，必须先说明偏离并将其视为阶段四工作统一整理。

进入阶段四前的强制说明：

- 阶段三结束时，必须先输出“编码完成总结”和“拟进入 UT 阶段”的说明，再开始 UT 设计。
- 未正式进入阶段四前，严禁修改 `tests/` 目录、mock/stub 文件、兼容性检查文件以及 `docs/ut_design.md`。

## 阶段四：单元测试

### 目标

本阶段分两步，但仍属于同一阶段：

1. 先输出单元测试设计文档 `docs/ut_design.md`，包括 UT 覆盖场景设计、测试点与实现映射、拟修改的 UT 文件列表等。
2. 在**需要确认模式**下，必须等待用户确认 `docs/ut_design.md` 没有问题后，才能开始 UT 编码；在**无需确认模式**下，产出文档后可直接继续。
3. 根据 `docs/ut_design.md` 完成 UT 编码实现。
4. 需要确认模式下，再次请求用户确认当前 UT；无需确认模式下直接进入编译验证阶段。

### 硬性要求

- 本阶段开始前，若 `docs/ut_design.md` 不存在，必须先创建并完成设计，禁止直接写 UT。
- UT 设计未完成前，严禁修改任何测试源码文件。
- 在需要确认模式下，`docs/ut_design.md` 未经用户明确确认前，严禁修改任何测试源码文件、mock/stub 文件、兼容性校验文件。
- 本阶段默认只修改测试相关文件和必要的测试依赖；若发现必须回改生产代码，必须明确说明原因，并在流程上回到阶段三处理，而不是在阶段四静默混改。

### 第一步：UT 设计必须覆盖

- 正常路径
- 参数非法路径
- 错误码返回
- 芯片差异路径
- 新增枚举/结构体兼容性
- 特性开关
- 驱动依赖相关分支

### 单元测试 → 编译验证：进入标准

1. `docs/ut_design.md` 已完成。
2. UT 编码已完成。
3. 满足以下其一：
   - 用户已明确确认 UT 设计和 UT 实现没有问题
   - 当前为无需确认模式
4. 不存在待修复的关键测试问题。
5. 用户没有明确限制“本次不做编译验证”。
6. Claude 已先输出 UT 设计总结与 UT 实现总结，而不是写完测试后直接开始编译。

进入阶段五前的强制说明：

- 未正式进入阶段五前，严禁运行构建、UT 编译、全量编译或其他编译验证命令。
- 若只是为了快速验证一处 UT 改动，也必须先满足本节进入条件。

## 阶段五：编译验证

### 目标

编译并修复所有错误，直到相关 target 均通过。编译通过后，必须自动进入代码检视阶段。

### 编译命令

```bash
# 1. 构建
bash build.sh

# 2. runtime UT 编译验证
export ASAN_OPTIONS=handle_segv=0:detect_leaks=0
bash tests/build_ut.sh --ut=runtime --target=runtime_ut -j2 -c

# 3. acl UT 编译验证
export ASAN_OPTIONS=handle_segv=0:detect_leaks=0
bash tests/build_ut.sh --ut=acl --target=ascendcl_utest -j2 -c
```

### 规则

- 编译失败时：分析错误、修复代码或测试、重新编译，直到通过。
- 编译成功时：
  - 输出编译结果与本轮验收范围。
  - 自动进入阶段六。

### 阶段五 → 阶段六：进入补充约束

- 只有在本轮约定范围内的编译或验证命令已经完成，且结果已对用户说明后，才能进入阶段六。
- 若阶段五中为修复编译问题新增了代码或 UT 修改，这些修改必须纳入阶段六检视范围，不能因“只是编译修复”而跳过检视。

## 阶段六：代码检视

### 目标

对已完成并通过编译验证的修改进行代码检视。支持两种模式：

1. 默认检视本轮编码实现、单元测试产生的修改。
2. 若用户提供 commit id，则对指定 commit 进行检视；若用户要求结合本轮修改，则两者都要覆盖并区分说明。

检视完成后，必须输出结构化检视意见文档 `docs/code_review.md`。若用户接受检视意见，或当前为无需确认模式，则在本阶段内按意见修改代码或 UT，并重新执行必要的 UT 补充判断、编译验证和再次检视，直到本阶段结束后不再产生任何代码修改。

### 检视范围

默认覆盖：

- 阶段三代码修改
- 阶段四 UT 修改
- 阶段五为修复编译问题产生的修改

### 检视维度

至少覆盖：

1. 规范性
   - 命名规范
   - 注释完整性
   - 代码风格
   - 是否符合 `.clang-format`
   - 是否引入无关改动
2. 合理性
   - 逻辑是否正确
   - 分层是否合理
   - 参数校验是否完备
   - 错误处理是否恰当
   - 是否符合 `implementation_plan.md`
3. 单元测试充分性
   - 是否缺失关键正常、异常、芯片差异、兼容性场景
   - 是否需要新增或调整测试文件
4. 可优化空间
   - 性能
   - 可读性
   - 可维护性
   - 可扩展性

### 检视意见文档至少包含

1. 检视范围
2. 总体结论：通过 / 有条件通过 / 不建议合入
3. 问题清单：
   - 问题级别
   - 问题位置
   - 问题描述
   - 修改建议
   - 是否需要补 UT
4. 单元测试补充建议
5. 后续动作建议

## 格式化约束

对所有修改的代码行，严格遵守以下规则：

1. 使用仓库根目录 `.clang-format` 进行格式化：`clang-format -style=file`
2. 只格式化本次修改后的代码行或代码块，不做全文件格式化
3. 禁止引入与本次需求无关的纯格式化噪声
4. 严禁修改原有行尾符号，保留文件现有 CRLF 或 LF

## 结束条件

仅当代码检视阶段完成后满足以下条件，整个流程才算结束：

1. 不再产生新的代码修改
2. 不再需要补充 UT
3. 不再需要补充或修正文档
4. 当前模式要求下的确认动作已完成
